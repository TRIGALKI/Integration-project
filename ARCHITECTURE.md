Сервисная архитектура и оркестрация
1. Архитектура сервиса: Монолитный подход

В рамках данного учебного проекта выбран монолитный архитектурный подход.

Обоснование:

Простота реализации и поддержки: Сервис реализует одну,
четко очерченную бизнес-задачу — предоставление REST API для предсказания вероятности дефолта. 
Вся логика (загрузка моделей, предобработка данных, эндпоинты API) инкапсулирована в одном приложении. 
Это упрощает разработку, отладку и тестирование, так как не требует настройки сетевого взаимодействия между сервисами.
Низкая операционная сложность: 
Для запуска и мониторинга требуется управлять только одним процессом.
Это снижает накладные расходы на инфраструктуру (например, не нужен Service Discovery или API Gateway).
Отсутствие необходимости в независимом масштабировании: 
В текущей постановке задачи нет требований к независимому масштабированию компонентов 
(например, чтобы предобработка данных работала на одних серверах, а предсказание — на других).
Горизонтальное масштабирование самого монолита (запуск нескольких инстансов за балансировщиком) 
полностью покрывает базовые потребности в отказоустойчивости и производительности.
Сложность данных: Модель машинного обучения и её веса являются ядром сервиса.
Разделение логики на микросервисы (например, отдельный сервис для predict и отдельный для health) 
не принесет архитектурных преимуществ, но усложнит деплой и синхронизацию версий моделей.

Микросервисный подход был бы оправдан, если бы сервис предсказания дефолта являлся лишь одной из частей 
сложной системы (например, с отдельными сервисами для сбора данных, обучения моделей, управления пользователями),
где разные компоненты имеют разные требования к ресурсам и частоте обновлений.

2. Концепт брокеров сообщений (на примере RabbitMQ)

В гипотетическом сценарии масштабирования, когда количество входящих запросов на предсказание резко возрастает или требуется обработка тяжелых батч-запросов, монолитная архитектура может стать узким местом. Для решения этой проблемы можно внедрить брокер сообщений, например, RabbitMQ.

Архитектура с брокером сообщений:

Веб-сервис (Producer): Принимает HTTP-запрос от клиента. 
Вместо того чтобы сразу выполнять ресурсоемкую операцию предсказания, он сериализует запрос (например, в JSON) 
и отправляет его в очередь RabbitMQ. Ответ клиенту формируется асинхронно 
(например, через WebSocket, long-polling или отдельный эндпоинт статуса).
Очередь (Queue): RabbitMQ выступает в роли буфера. 
Он накапливает входящие задачи на предсказание. Это позволяет веб-сервису выдерживать пиковые нагрузки, 
не "падая" от перегрузки, так как он лишь быстро ставит задачи в очередь.
Сервисы-обработчики (Consumers): Отдельные worker-процессы (или пул воркеров) в фоновом режиме забирают задачи из очереди,
выполняют предсказание с помощью модели и отправляют результат в базу данных или другую очередь для отправки
уведомления клиенту.

Для чего это может пригодиться:

Асинхронная обработка батч-предсказаний: Клиент может отправить запрос на предсказание для 10 000 клиентов. Веб-сервис мгновенно вернет task_id, а воркеры будут постепенно обрабатывать этот батч в фоновом режиме.
Декаплинг компонентов: Веб-сервис не зависит от доступности модели и вычислительных мощностей воркеров. Если модель временно недоступна или воркеры перезапускаются, запросы просто накапливаются в очереди.
Логирование и аудит: Все входящие запросы можно дублировать в специальную очередь логов. Отдельный сервис-логгер будет забирать сообщения из этой очереди и записывать их в централизованное хранилище (например, Elasticsearch), не создавая лишней нагрузки на основной воркер.
3. Логирование и мониторинг

Логирование в коде:
В коде используется стандартный модуль logging из Python.

Формат: По умолчанию логи выводятся в текстовом формате в стандартный вывод (stdout). 
Однако для production-среды настоятельно рекомендуется использовать формат JSON. 
Это легко настраивается через logging.Formatter или с помощью сторонних библиотек (например, python-json-logger). JSON-логи легко парсить автоматическими системами.
Куда логируются: В режиме debug=False при запуске через gunicorn или другой WSGI-сервер в контейнере Docker 
логи будут выводиться в stdout. 
Это является стандартной практикой для облачных платформ (Kubernetes, Docker Swarm), 
которые автоматически собирают логи со стандартного вывода контейнеров.

Сбор и мониторинг логов в production-системе:
Для централизованного сбора, хранения, анализа и визуализации логов используется стек ELK (Elasticsearch, Logstash, Kibana) или его аналоги.

Сбор: Агент на каждом хосте собирает логи из файлов или со стандартного вывода контейнеров и отправляет их в Logstash.
Обработка: Logstash принимает поток логов, парсит JSON-структуру и обогащает данные (добавляет теги, метки окружения).
Хранение и поиск: Обработанные данные сохраняются в Elasticsearch — мощном поисковому и аналитическому движку.
Визуализация: Kibana подключается к Elasticsearch и предоставляет удобный веб-интерфейс для создания дашбордов. На них можно отслеживать:
Количество запросов по эндпоинтам (/predict, /health).
Частоту ошибок (400, 500).
Время ответа сервиса.
Статусы загрузки моделей при старте приложения.

Такая связка позволяет инженерам оперативно реагировать на инциденты, анализировать поведение системы под нагрузкой и отслеживать ключевые метрики производительности (KPI).
