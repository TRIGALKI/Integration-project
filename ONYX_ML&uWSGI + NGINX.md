# Оптимизация модели: Преобразование в ONNX-ML

## ONNX (Open Neural Network Exchange) — это открытый формат для представления моделей машинного обучения. Он позволяет моделям, обученным в одном фреймворке (например, scikit-learn, PyTorch, TensorFlow), быть использованными в другом.

## ONNX-ML — это расширение стандарта ONNX, специально разработанное для классических моделей машинного обучения (из scikit-learn, LightGBM, XGBoost и др.).

Преимущества использования ONNX:
Ускорение инференса: Движки для выполнения ONNX-моделей (например, ONNX Runtime) оптимизированы и могут выполнять предсказания значительно быстрее, чем стандартные библиотеки Python.
Кроссплатформенность: Модель можно легко развернуть в различных средах (C++, C#, Java).
Упрощение MLOps: Единый формат модели упрощает процесс её хранения, версионирования и развертывания.
Для конвертации используется библиотека skl2onnx.

**Пример конвертации LogisticRegression:**

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxruntime as rt

# Конвертируем Pipeline в ONNX
initial_type = [("float_input", FloatTensorType([None, n_features]))]
onnx_model = convert_sklearn(pipeline, initial_types=initial_type)

with open("models/model_v1.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())

# Инференс через ONNX Runtime
sess = rt.InferenceSession("models/model_v1.onnx")
pred = sess.run(["output_label"], {"float_input": X_test.astype("float32")})
```

Production-развертывание: uWSGI + NGINX

Запуск Flask-приложения через встроенный сервер (app.run()) подходит для тестов но не подходит для production-среды. Он не рассчитан на высокую нагрузку, не поддерживает многопоточность и небезопасен. Для промышленного использования Flask-приложение "оборачивают" в более производительный сервер приложений и ставят перед ним веб-сервер.

## NGINX — это высокопроизводительный веб-сервер и обратный прокси-сервер.

Обработка статических файлов: NGINX напрямую, без участия Python, 
отдает статический контент (картинки, CSS, JavaScript), что сильно разгружает приложение.
Балансировка нагрузки (Load Balancing): NGINX может распределять входящие запросы
между несколькими экземплярами вашего Flask-приложения (работающими на разных портах или даже разных серверах).
Терминация SSL/TLS: NGINX берет на себя ресурсоемкую задачу по шифрованию и дешифрованию HTTPS-трафика, 
передавая внутрь уже расшифрованные HTTP-запросы.
Ограничение частоты запросов (Rate Limiting): Защищает бэкенд от DDoS-атак и резких всплесков трафика.
Обратный прокси (Reverse Proxy): NGINX принимает все входящие запросы и перенаправляет их 
на сервер приложений (uWSGI).

## uWSGI — это сервер приложений, который реализует протокол WSGI (Web Server Gateway Interface).
## Он является "мостом" между NGINX и Python-кодом.

Высокая производительность: uWSGI написан на C и оптимизирован для быстрой обработки запросов.
Управление процессами: uWSGI управляет пулом воркеров (рабочих процессов) вашего приложения. 
Он может автоматически перезапускать их при падении, управлять их количеством в зависимости от нагрузки.
Позволяет запускать несколько процессов и потоков для одновременной обработки множества запросов.
Изоляция: Работает как отдельный сервис, что повышает стабильность и безопасность системы.
Схема работы
Клиент отправляет запрос (например, GET /predict).
Запрос поступает на NGINX.
NGINX определяет, что запрос динамический (не статический файл), 
и по настроенным правилам проксирует его на uWSGI (например, на localhost:8000).
uWSGI получает запрос, выбирает свободный воркер из своего пула и передает запрос вашему Flask-приложению.
Flask-приложение обрабатывает запрос и возвращает ответ.
Ответ передается обратно через uWSGI в NGINX, который отправляет его клиенту.
